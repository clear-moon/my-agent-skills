# GitHub Upload Merge Skill

## 技能目标

这个技能用于把“给定一个 GitHub 仓库地址 + 一个本地目录”的重复流程固化成稳定步骤，减少每次手工判断远程、分支、提交、合并和清理的心智负担。

默认目标不是只把文件推上去，而是完整做完这一条链路：

1. 找到或克隆目标仓库
2. 检查远程和工作区状态
3. 新建功能分支
4. 把指定目录同步到目标仓库
5. 提交并推送
6. 在本地合并到 `main`
7. 推送 `main`
8. 删除本地和远程功能分支

## 何时使用

仅当用户明确要求使用本技能，或明确表示按这套完整流程执行时，才使用本技能。典型情况包括：

- 明确要求“使用 `github-upload-merge` 技能”
- 明确表示“按这个技能的完整流程执行”
- 已提供 GitHub 仓库地址和本地目录，并明确要求你完整执行上传、提交、推送、合并与清理
- 明确要求“本地合并 `main`”
- 明确要求“合并后删本地和远程分支”
- 想验证某个目录发布到 GitHub 的完整流程是否打通

不适合使用本技能的情况：

- 用户只是想查看仓库内容，还没有要求上传
- 用户没有给出仓库地址，也没有给出本地目录
- 目标不是 GitHub 仓库
- 用户只是泛泛地说“帮我上传到 GitHub”，但没有明确要求使用本技能或这套完整流程
- 用户只想生成文档、草稿或命令说明，不希望实际执行

## 输入要求

至少确认以下信息：

- `repo_url`：目标 GitHub 仓库地址
- `local_path`：需要上传的本地目录

如果以下信息缺失，需要先补齐或推断：

- `repo_local_path`：目标仓库在本机的本地路径；如果没有现成克隆，可自行克隆
- `target_subpath`：目录上传到仓库中的目标位置；若用户未指定，优先询问，或在明显只有一种合理位置时说明默认值后执行
- `branch_name`：功能分支名；未指定时根据内容生成简洁稳定的分支名

## 默认执行策略

本技能在被显式调用后，默认自动执行完整流程，不在 `push` 或 `merge` 前暂停。

但出现以下情况时，必须先停下并询问用户，而不是继续硬做：

- 当前工作区不干净，且现有改动与你要处理的内容无关
- 目标仓库没有写权限或认证不可用
- 目标路径不明确，可能覆盖错误目录
- 合并 `main` 前出现冲突
- 远程分支保护策略阻止直接推送 `main`
- 本地没有目标仓库，且你无法安全确认克隆位置

## 标准流程

### 1. 预检查

先确认运行环境和基础条件：

- 当前系统类型，Windows 优先使用 PowerShell 命令
- 本地目录是否存在
- 目标仓库是否已在本地克隆
- 目标仓库远程地址是否与 `repo_url` 一致
- GitHub 认证或 Git 推送能力是否可用

如果会启动新的长流程或疑似已有相关终端，先检查现有终端状态，避免重复操作。

### 2. 定位目标仓库

优先使用本机已有克隆仓库，避免重复 clone。

若本地没有现成仓库：

1. 先确认父目录存在且位置合理
2. 将仓库克隆到明确路径
3. 进入该仓库继续后续步骤

### 3. 检查 Git 状态

至少检查：

- `git status --short --branch`
- `git remote -v`

必要时补充：

- 当前分支是否跟踪远程
- `main` 是否存在并可拉取
- 工作区是否干净

如果仓库中已有不相关改动，不要覆盖或回退它们；应暂停并说明风险。

### 4. 创建功能分支

分支命名应简洁且可读，通常使用：

- `feat/add-<name>`
- `chore/upload-<name>`

如果用户已经指定分支名，优先使用用户提供的名称。

### 5. 同步目录到仓库

把 `local_path` 中需要上传的内容同步到目标仓库的 `target_subpath`。

执行前要先确认：

- 目标父目录存在
- 是否会覆盖已有文件
- 是否需要保留目录名本身，还是只复制目录内容

如果用户没有说明，应在执行前明确你采用的是哪种同步方式。

### 6. 审查改动

在提交前至少查看：

- `git status --short --branch`
- `git diff`
- 最近提交信息风格

提交信息要反映“为什么要上传这批内容”，不要只机械描述“update files”。

不要提交看起来像密钥、凭证或环境变量的敏感文件。

## 7. 提交与推送

按顺序执行：

1. `git add` 相关文件
2. `git commit`
3. `git push -u origin <branch>`

如果推送失败，先判断是认证问题、权限问题还是远程保护策略问题，再决定是否继续。

### 8. 本地合并 main

按固定顺序执行：

1. `git checkout main`
2. `git pull origin main`
3. `git merge <feature_branch>`
4. `git push origin main`

如果仓库默认分支不是 `main`，先确认再执行，不要盲猜。

如果 `merge` 产生冲突，停止自动流程，向用户报告冲突位置和下一步建议。

### 9. 删除分支

合并成功后执行：

- 删除本地功能分支
- 删除远程功能分支

只有在确认变更已经进入远程 `main` 后，才删除功能分支。

## 输出模板

完成后按以下结构向用户汇报：

### 执行结果

- 目标仓库：`<repo_url>`
- 本地仓库：`<repo_local_path>`
- 上传目录：`<local_path>`
- 目标位置：`<target_subpath>`
- 功能分支：`<branch_name>`
- 提交结果：`<commit_summary>`
- 合并状态：`已合并到 main / 未合并`
- 清理状态：`本地分支已删除 / 远程分支已删除 / 未删除及原因`

### 验证建议

- 检查远程 `main` 是否已包含目标目录
- 检查目标仓库页面是否能看到新文件
- 如有 README 或索引要求，确认是否同步更新

## 风险处理原则

- 不要改动与本次上传无关的现有修改
- 不要使用破坏性 Git 命令清理用户现场
- 不要默认强推
- 不要在分支保护阻止时绕过规则
- 一旦发现异常变更、冲突或目标路径不明，立刻停下并询问

## 示例提示词

**示例 1**

请使用 `github-upload-merge` 技能，把 `C:\Users\me\Desktop\demo-skill` 上传到 `https://github.com/me/my-agent-skills.git`，按标准流程新建分支、提交、推送、本地合并 `main`，最后删除本地和远程分支。

**示例 2**

我给你一个 GitHub 仓库地址和一个本地目录，请按 `github-upload-merge` 这套完整流程执行。如果本地已经有克隆仓库就复用，没有就自己 clone，最后用本地方式合并进 `main`。

**示例 3**

请按 `github-upload-merge` 技能的完整流程，把这个本地目录发布到我自己的 GitHub 仓库里：检查远程、建分支、push、merge main、删分支。
